Архитектура мультикуба M03.01 Импорт факт заказов

Описание архитектуры

1. Источник данных (SQL Database)

Данные загружаются из MS SQL базы данных из таблицы CF_Import_By_Orders с фильтрацией по периодам, заданным в мультикубе М94.03 Фильтр периодов загрузки Заказов.

Загружаемые поля из SQL:

  • PERIOD → Период
  • AUFNR → Заказ
  • KSTAR → Вид затрат
  • PostDate → Дата проводки
  • Quantity → Общее ВведеннКолич
  • UoM → ВведЕдиницИзмерен
  • MatNum → Материал
  • MatKText → Краткий текст материала
  • DocComment → Название
  • Description → Описание корреспондирующего счета
  • RepSum → Знач/ВлтОтч
  • RepCurr → Валюта отчета
  • LinkDocNum → № СсылочнДокумента
  • _User → Имя пользователя
  • DocData → Дата документа
  • BELNR → № документа
  • ObjSum → Сумма/ВлтОб
  • ObjCurr → Валюта объекта
  • HeadText → Текст заголовка документа
  • DocDataEntry → Дата ввода документа
  • DocTime → Время ввода документа

2. Обработка данных в мультикубе M03.01

Прямая загрузка (без формул):

Поля заполняются напрямую из SQL через скрипт импорта.

Расчетные поля (с формулами):

A. Маппинг и преобразование данных:

  • Item.Внутренний заказ - маппинг из справочника L01.08 Внутренние заказы по полю "Заказ"
  • Item.Вид затрат - маппинг из справочника L01.07 Виды затрат по полю "Вид затрат"
  • Item.МВЗ - получение через маппинг M97.02 Маппинг Заказ - МВЗ
  • Item.L04.02 1D Ответственные-Внутренние заказы - связь ответственных лиц с заказами

B. Контроль качества данных:

  • Check Внутренний заказ - проверка наличия заказа в справочнике (если есть "Заказ", но нет "Item.Внутренний заказ" → ошибка)
  • Check Вид затрат - проверка наличия вида затрат в справочнике
  • не проставлен ответственный - проверка наличия ответственного лица
  • Check_Кол-во ошибок - суммарное количество ошибок
  • Check_Описание ошибки - текстовое описание найденных ошибок

C. Фильтры и агрегация:

  • Есть данные - проверка наличия данных по периоду
  • NumCount - подсчет количества записей с данными
  • Есть ошибка - флаг наличия ошибок в данных

3. Использование данных (Referenced By)

Данные из M03.01 используются в:

  • M01.04 Ввод данных (по заказам) - поле 'Ввод без факта' использует данные из M03.01
  • M99.01 MaxNumImport - поле 'MsxNumOM Заказ' для отслеживания максимального номера импорта
  • М11.02 1D Ответственные-Внутренние заказы - использует связку ответственных и заказов

4. Вспомогательные компоненты:

  • М94.03 Фильтр периодов загрузки Заказов - управляет периодами для загрузки
  • M98.01 Настройки скриптов - хранит параметры подключения к SQL
  • M99.02 Script logging - логирование выполнения скрипта
  • M97.02 Маппинг Заказ - МВЗ - справочник соответствия заказов и МВЗ

Ключевые особенности архитектуры:

  1. Трехуровневая структура:

    • Уровень 1: Источник (SQL DB) + Настройки
    • Уровень 2: Обработка и валидация (M03.01)
    • Уровень 3: Использование в аналитических кубах

  2. Контроль качества данных встроен непосредственно в мультикуб через систему проверок (Check-поля)

  3. Маппинг данных происходит через справочники и специализированные кубы маппинга

  4. Логирование всех операций в отдельный мультикуб для аудита

  5. Фильтрация периодов управляется отдельным мультикубом, что позволяет гибко настраивать загрузку данных

# Архитектура импорта «Факт заказов» (читабельная версия)

1) Назначение

Процесс загружает фактические данные по заказам из MS SQL в центральный мультикуб M03.01, обогащает их справочниками/маппингами, выполняет проверки качества и передаёт результат в целевые мультикубы для план/факт, контроля импорта и сводной витрины ошибок.

2) Компоненты (что участвует)

2.1 Источники и настройки

Тип Код/имя Роль
SQL-источник MS SQL / CF_Import_By_Orders Таблица-источник факта заказов
Управление периодами M94.03 Фильтр периодов загрузки Заказов Задает список периодов для WHERE PERIOD IN (…)
Настройки подключения M98.01 Настройки скриптов SQL Host/Port/User/Password/DB + параметры ядра/логгера
Логирование M99.02 Script logging DEBUG-лог выполнения скрипта импорта
Скрипт JavaScript (Импорт факт заказов) Extract/Load из SQL в M03.01

3) Центральный узел: M03.01 Импорт факт заказов

3.1 Структура M03.01 по блокам (логика «от сырья к обогащению»)

Блок 1 — Прямая загрузка из SQL (без формул)

Источник: CF_Import_By_Orders
Сущность: «как в SQL» (сырые поля).
Примеры полей: Период, Заказ, Вид затрат, Дата проводки, Материал, Валюты/суммы, № документа, пользователь, даты/время ввода и т.п. (≈21 поле).

Блок 2 — Маппинг основных сущностей (ключевые “Item.*”)

Цель: превратить текст/коды из SQL в элементы справочников/кубов.

Выходное поле M03.01 Как вычисляется Откуда берётся
Item.Внутренний заказ FINDBYNAME('L01.08 Внутренние заказы','Заказ') справочник L01.08
Item.Вид затрат FINDBYNAME('L01.07 Виды затрат','Вид затрат') справочник L01.07
Item.МВЗ SELECT('M97.02 Маппинг Заказ - МВЗ','Item.Внутренний заказ') маппинг-куб M97.02
Item.L04.02 Ответственные-Внутренние заказы SELECT('M11.02 1D Ответственные-Внутренние заказы', …) маппинг-куб M11.02

Критично: Item.Внутренний заказ — корневой ключ. От него зависит большая часть обогащения.

Блок 3 — Атрибуты заказа (через свойства L01.08)

Все атрибуты подтягиваются через:
PROPERTY('L01.08'.'<свойство>', 'Item.Внутренний заказ')

Выходные атрибуты (примерно 9):

  • Вид контракта, Вид работ, Статус заказа, Группа заказов, Проект, Тип заказа, Объект учета затрат, Контракт, Название объекта (Описание)

Блок 4 — Атрибуты МВЗ (через свойства L01.04 МВЗ)

PROPERTY('L01.04 МВЗ'.'<свойство>', 'Item.МВЗ')

Выходные атрибуты (6):

  • Вид деятельности, Подразделение, Филиал, Регион, ЦФО, Направление

Блок 5 — Атрибуты видов затрат

Выходное поле Формула Источник
Item.Вид затрат (Сводная) PROPERTY('L01.07'.'Вид затрат (Сводная)', 'Item.Вид затрат') L01.07
Обозначение В/Затрат PROPERTY('L01.03'.'Обозначение В/Затрат', 'Item.Вид затрат') L01.03

Блок 6 — Контроль качества (валидаторы)

Проверка Логика Назначение
Check Внутренний заказ IF BOOLEAN('Заказ') AND NOT BOOLEAN('Item.Внутренний заказ') THEN 1 ELSE 0 не найден заказ в L01.08
Check Вид затрат IF BOOLEAN('Вид затрат') AND NOT BOOLEAN('Item.Вид затрат') THEN 1 ELSE 0 не найден вид затрат в L01.07
не проставлен ответственный IF BOOLEAN('Item.Внутренний заказ') AND NOT BOOLEAN('Item.L04.02...') THEN 1 ELSE 0 предупреждение по ответственным
Check_Кол-во ошибок сумма ключевых check-полей агрегатор ошибок
Check_Описание ошибки текстовая сборка по активным check человекочитаемый лог
Есть ошибка Check_Кол-во ошибок > 0 флаг записи с ошибкой

В текущей логике «не проставлен ответственный» — отдельный контроль (часто как предупреждение), а в Check_Кол-во ошибок обычно суммируются именно ошибки маппинга (заказ/вид затрат).

Блок 7 — Служебные поля

  • Есть данные = наличие периода
  • NumCount = 1/0 для подсчёта записей
  • sNum ActImportOrders = нумерация записей по периодам (служебно для импорта/контроля)

5) Поток данных (ETL) — кратко и по шагам

5.1 Шаги обработки

  1. Extract:

    • прочитать M98.01 (SQL параметры)
    • прочитать M94.03 (периоды)
    • выполнить SELECT * FROM CF_Import_By_Orders WHERE PERIOD IN (...)
    • логировать в M99.02

  2. Transform (внутри M03.01):

    • загрузить «сырые» поля
    • выполнить FINDBYNAME/SELECT/PROPERTY
    • посчитать check-поля и служебные поля

  3. Load / использование:

    • M03.01 как витрина факта + качества
    • передача в потребителей (M01.04, M99.01, M99.06)

8) Критические зависимости (коротко)

  • Item.Внутренний заказ — критическая точка: от него зависят атрибуты заказа, получение МВЗ, ответственных и косвенно орг-атрибуты.
  • Item.Вид затрат — важная точка для аналитики и агрегирования.
  • Подключение к SQL — критично для всего процесса; диагностика через M99.02.

9) Мини-набор KPI (для контроля качества)

  • Успешность маппинга заказов = доля записей с Check Внутренний заказ = 0
  • Успешность маппинга видов затрат = доля записей с Check Вид затрат = 0
  • Общее качество импорта = доля записей с Check_Кол-во ошибок = 0
  • Полнота ответственных = доля записей без не проставлен ответственный
