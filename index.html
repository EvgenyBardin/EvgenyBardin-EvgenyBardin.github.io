Как проанализировать текущие настройки SAP R/3 (Production & Production‑Controlling) без документации

1️⃣ Собрать «сырой» материал о системе

Что собрать Где искать (транзакции/таблицы) Как выгрузить
Структура IMG‑настроек SPRO → SAP Reference IMG Экспорт в PDF/HTML через SAP Solution Manager → Project → Documentation или ручной скрин‑шот + RZ11‑RZ10‑RZ13 (параметры)
Список всех пользовательских объектов (Z‑разработки) SE80, SE84, SE38, SE24 → Таблица TADIR (поле OBJ_NAME = ‘Z*’) SE16N → TADIR → выгрузка в CSV/Excel
Исходный код ABAP‑объектов SE38/SE80 (программы, функции, классы) RS38M – Download ABAP source (в XML/ZIP)
Таблицы‑конфигурации производственного модуля PP – T001W, T024, T024D, T156, T156T, T156S, T156U, T156V, T156W, T156X, T156Y, T156Z 
CO‑PC – COSP, COSS, COEP, COBK, COBJ, COVP		 SE16N → Export
Настройки BOM/Production Versions Таблицы STKO/STPO (BOM), MAST (BOM‑usage), AUFK/AFKO/AFVC (производственные заказы) SE16N → Export
Пользовательские отчёты/queries SQ01/SQ02/SQ03, SE38 (report) Выгрузка кода и/или XML‑описания
Workflow/Business‑Process‑Models SWDD (workflow), SWDD → SWDD‑Export SWDD → Export
_Настройки Cost‑Estimate (CK)_* Таблицы CKMLV, CKMLV2, CKIS, CKISV SE16N → Export
Транзакционные логи (Change‑Document) CDHDR/CDPOS (изменения в мастерах) SE16N → Export
Событийные логи (Production Order History) Таблицы AFRU, AFVC, AFVV, AFVC SE16N → Export
Пользовательские настройки в SAP Solution Manager SOLMAN → Configuration Export via Solution Documentation

Почему именно эти данные?

  • Они покрывают всё, что обычно «скрыто» в настройках PP/CO: структуры BOM, версии, тарифы, цены, контроллинговые счета, правила оценки стоимости.
  • Таблицы TADIR и TADIR‑OBJ_NAME='Z*' дают полный список всех кастомных объектов – их нельзя пропустить.
  • CDHDR/CDPOS и логи заказов позволяют увидеть, как реально пользователи используют систему (что важнее, чем только конфигурация).

2️⃣ Техническая подготовка выгрузки

  1. Создать отдельный “Export‑Server” (можно Linux/Windows VM) с установленным SAP GUI и SAP RFC‑SDK.
  2. Написать простой ABAP‑report (или использовать готовый RZ20‑RZ20‑RZ20) который:
    • Выбирает нужные таблицы (см. пункт 1).
    • Пакует их в ZIP‑архив (каждая таблица – отдельный CSV/JSON).
    • Сохраняет в application server (директория /usr/sap/tmp/export/).
  3. Скопировать архив на аналитический сервер (SCP/FTP).

Альтернатива – использовать SAP Data Services / SAP BW (если есть) для автоматической репликации данных в отдельную базу (например, PostgreSQL) и дальше работать уже с ней.

3️⃣ Построить «знание‑базу» (knowledge‑base) из выгруженных артефактов

Шаг Инструмент Что делаем
3.1 ElasticSearch / OpenSearch (или Pinecone, Weaviate) Индексируем все текстовые файлы (ABAP‑source, CSV‑описания таблиц, IMG‑PDF).
3.2 Embedding‑модель (OpenAI text-embedding-ada-002, HuggingFace sentence‑transformers/all‑mpnet‑base‑v2) Преобразуем каждый документ/строку в вектор.
3.3 Vector DB – сохраняем вектор + метаданные (имя файла, тип, ссылка на оригинал).
3.4 LLM‑интерфейс – подключаем ChatGPT‑4 (OpenAI), Claude 3, LLaMA‑2‑Chat (если нужен локальный вариант).
3.5 Prompt‑шаблоны – задаём «system prompt», где описываем контекст: «Ты – эксперт по SAP R/3 PP/CO. Тебе доступна внутренняя база с исходным кодом, конфигурацией и таблицами. Отвечай, используя только данные из этой базы. Если чего‑то нет – укажи, что требуется дополнительный источник.»

Почему именно LLM?

  • Позволяет задать «естественный» вопрос («Какие шаги бизнес‑процесса по планированию производства?», «Где хранится тариф линии в текущей конфигурации?») и получить ответ, не разбираясь в структуре таблиц.
  • Можно быстро построить high‑level процессные карты (схемы) на основе найденных описаний.

4️⃣ Что именно загружаем в нейросеть (LLM)

Тип данных Формат Как подготовить
Текстовая документация IMG PDF → OCR → txt pdf2txt.py → чистый текст
ABAP‑source *.abap, *.txt Оставить оригинал, добавить комментарии (если есть)
Таблицы‑справочники CSV/JSON Сохранить заголовок + первые 50‑100 строк (чтобы LLM «видел» структуру)
Список Z‑объектов CSV (Obj_type, Obj_name, Package) Добавить поле Description (из SE11/SE80)
Change‑Document CSV Фильтровать только изменения в PP/CO (Obj_class = ‘PP’, ‘CO’)
Логи заказов CSV Выделить ключевые поля (AUFNR, MATNR, PLNNR, BATCH, COST)
Workflow‑описания XML/JSON Экспортировать из SWDD → Export
Конфигурационные параметры CSV (Param_name, Value) RZ11 → Export
Технические заметки (если есть) txt Скопировать из почтовых архивов, Confluence, Wiki

Важно: LLM не «видит» структуру БД, а лишь текст. Поэтому нужно добавить метаданные (таблица → описание полей) в каждый документ, чтобы модель могла понять смысл.

5️⃣ Получение верхнеуровневых бизнес‑процессов

  1. Запрос:

    text

    Опиши в виде BPMN‑диаграммы (или простого списка шагов) процесс «Планирование производства» в текущей системе SAP R/3, используя доступные данные.

  2. LLM ищет в базе:

    • Текст из IMG‑раздела Production Planning → Master Data → Material Master
    • Описание функций в ABAP‑модулях SAPLPP01, SAPLCO01
    • Таблицы AFKO, AFVC, AFVV (order header, operations)
    • Пользовательские Z‑объекты, которые могут расширять процесс (например, Z_PP_SCHEDULER).

  3. Ответ (пример):

    1. Создание/изменение мастер‑данных материала (MM01/MM02) – таблица MARA/MARC. 2. Формирование BOM (CS01) и определение Production Version (MAST). 3. Планирование спроса (MD61) → план потребностей (MD01/MD02) → генерируются плановые заказы (CO01). 4. Создание производственного заказа (CO01) → копирование BOM и Routings (AFKO/AFVC). 5. Расчет стоимости (CK40N) – использует цены из таблиц CKMLV (включая прогнозные цены из ZFCST_PRICES). 6. Выпуск заказа (COHV) → подтверждение операций (CO15) → расчёт фактической себестоимости (COSP/COSS). 7. Закрытие заказа (CO02) → перенос себестоимости в CO‑PC (COEP).

  4. При необходимости LLM генерирует PlantUML / Mermaid‑диаграмму, которую можно отобразить в Confluence или VS‑Code.

6️⃣ Альтернативные (не‑LLM) подходы для «видения» процессов

Подход Что нужен Плюсы Минусы
SAP Solution Manager (SolMan) – Documentation Assistant Доступ к системе SolMan Автоматически собирает IMG, Custom Code, Change‑Docs Требует лицензии и настройки SolMan
Process Mining (Celonis, ProcessGold, UiPath Process Mining) Event‑log из таблиц AFKO, AFVC, COSP Наглядные карты реального исполнения, узкие места Нужно собрать лог, лицензия
ABAP‑Doc / SAP Code Inspector Тексты ABAP, комментарии Быстро генерирует техническую документацию Не покрывает бизнес‑процессы
Enterprise Architecture Tools (Sparx EA, ARIS) + SAP‑Connector Метаданные SAP (RFC) Позволяют построить диаграммы компонентов Требует ручного моделирования

7️⃣ Пошаговый план внедрения «LLM‑ассистента»

Этап Действия Ожидаемый результат
1. Инвентаризация Сформировать список всех Z‑объектов (TADIR), собрать IMG‑PDF, выгрузить ключевые таблицы. Полный набор «сырого» контента (~200‑300 МБ).
2. Подготовка данных Привести файлы к единому формату (txt/CSV/JSON), добавить метаданные (таблица → описание полей). Структурированная коллекция, готовая к индексации.
3. Выбор векторного хранилища Deploy Weaviate (Docker) или Pinecone (cloud). Рабочий Vector DB.
4. Генерация эмбеддингов Запустить скрипт (Python) → openai.embeddings.create (или sentence‑transformers). Сохранить в DB. Векторные представления всех документов.
5. Интеграция LLM Создать API‑прослойку (FastAPI) → OpenAI GPT‑4 (или локальный Llama‑2‑Chat‑70B). При запросе: 1) ищем релевантные документы (top‑k ≈ 10) → 2) формируем контекст + пользовательский запрос → 3) отправляем в LLM → 4) возвращаем ответ. Интерактивный чат‑бот, отвечающий на вопросы о PP/CO.
6. Тестирование Сформировать набор вопросов: «Где хранится тариф линии?», «Какой отчёт используется для сравнения плановых и фактических затрат?», «Какие Z‑транзакции влияют на расчёт стоимости?». Оценка точности, доработка индекса (добавить недостающие файлы).
7. Визуализация процессов Добавить в чат‑бот возможность генерировать Mermaid‑диаграммы (LLM возвращает код, бот рендерит). Автоматическое построение схем процессов.
8. Документация и передача Сформировать «User‑Guide» по работе с чат‑ботом, описать ограничения, расписать процесс обновления данных (ежемесячный экспорт). Готовый инструмент для бизнес‑аналитиков и тех. специалистов.

8️⃣ Что ещё стоит учесть

  1. Безопасность – выгруженные данные могут содержать конфиденциальные цены, номера заказов, персональные данные. При загрузке в облако (OpenAI, Pinecone) необходимо анонимизировать:

    • Удалить поля KUNNR, LIFNR, MATNR (заменить на XXX).
    • Оставить только метаданные (названия полей, типы, бизнес‑логика).

  2. Версионирование – хранить выгрузки с датой (e.g., export_2024-04-14.zip). При каждом обновлении пересоздавать эмбеддинги, чтобы LLM видел актуальное состояние.

  3. Согласование с IT‑безопасностью – если планируется использовать облачный LLM, получить одобрение от CISO, оформить договор DPA.

  4. Пост‑обучение модели (опционально) – если есть достаточно примеров вопросов‑ответов, можно дообучить небольшую модель (e.g., Mistral‑7B‑Instruct) с помощью LoRA на ваш набор Q&A. Это уменьшит зависимость от внешних API.

  5. Периодический аудит – проверять, что новые Z‑объекты (разработанные после начала проекта) автоматически попадают в выгрузку (скрипт SELECT * FROM TADIR WHERE OBJ_NAME LIKE 'Z%' AND CHANGED_ON > last_run).

Что делаем Как делаем Инструменты

Эта цепочка позволяет получить верхнеуровневое представление бизнес‑процессов (планирование, производство, контроль себестоимости) без наличия официальной документации, используя только то, что уже живёт
